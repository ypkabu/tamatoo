## TL;DR

- スペインの「La Tomatina」をモチーフにした **2 人協力体験型ゲーム** を Unreal Engine 5.7 で個人/小規模チーム制作した。
- **1P=カメラマン、2P=タオル拭き取り**。1P はモニターを見てトマトを発射、2P は LeapMotion で手をかざしてメイン画面の汚れを拭き取る。**スマホ画面（縦長サブディスプレイ）には“被写体のズーム映像”が同時に映る**。
- 一番の山は描画構成。当初は **1 つのウィンドウ内に Widget でスマホ枠を表示** していたが、解像度・DPI・モニター構成が変わるたびに引き伸ばし・座標ズレが起きた。これを **`SWindow` を `FSlateApplication` に追加してデュアルウィンドウ化** することで「解像度変数を 1 つ変えるだけ」で済む構成に作り替えた。
- 派生して、UE 5.7 特有の **SceneCapture が自動キャプチャされない問題**、**第二ウィンドウで RenderTarget が再評価されない Slate の挙動**、**ズームカメラの壁抜け / 空ヒット** などにぶつかり、それぞれコードレベルで対処した。
- 試遊会フィードバックとして **「1P がスマホ画面を見ながら操作するのが難しい」** という UX の課題が残っており、本記事の最後で正直に振り返る。

このゲームは「動いた / 完成した」だけでなく、**“設計をやり直して負債を畳んだ”プロセス自体が一番見せたい価値** だと考えています。

## 1. このゲームは何か

### 1.1 コンセプト

- 元ネタ: スペイン・ブニョール「La Tomatina」（毎年 8 月最終水曜、街中でトマトを投げ合う祭り）
- ゲームの体験は **「祭りに参加した友達 2 人」** をそのまま操作に置き換えた:
  - 1P は **カメラマン**: トマトを街にぶつけて、汚れた様子を“撮る”
  - 2P は **巻き込まれる側 / 後始末**: タオルで画面の汚れを拭き取る
- 1P と 2P は **同じスクリーンを共有しない**。1P はメインモニター、2P は LeapMotion を覗き込みながらメイン画面に手をかざす。**「視線と手が交差する」物理的なレイアウト** を狙っている。

### 1.2 ハード構成（実際の設置）

```
              ┌───────────────────┐
              │   メインモニター     │ ← 1P が見る・トマト発射
              │  (ゲーム本編 16:9) │ ← 2P がここに手をかざす
              └─────────┬─────────┘
                        │
   ┌─────────────────┐  │   ┌──────────────────┐
   │ スマホ画面 (縦長) │ ← │ → │  LeapMotion       │
   │ ズーム映像表示    │     │   (2P 用ハンド入力) │
   └─────────────────┘     └──────────────────┘
        ↑
     1P が覗き込んで構図を確認
```

- **メインモニター**: 通常のゲーム画面。三人称視点で街を見下ろす。
- **スマホ画面（実機を縦置き）**: PC の拡張ディスプレイとして接続。`RT_Zoom` という RenderTarget の中身がここにフルスクリーン表示される。1P から見ると **「自分が構えているスマホのカメラを覗いている」** ように見える。
- **LeapMotion**: メインモニターの前下方に設置。2P が手をかざすと、その位置がメイン画面上の正規化座標に変換され、`UTomatinaTowelSystem` 経由で `ATomatoDirtManager::WipeDirtAt` を呼んで汚れを薄くする。

### 1.3 1 ラウンドの流れ（ゲームループ）

1. ミッション提示（例:「赤トマトを N 個当てろ」「黄色いトマトを連続ダッシュで剥がせ」）
2. 1P がトマトを発射 → 着弾位置に汚れ（`FDirtSplat`）が追加される
3. 2P が LeapMotion で手を動かす → 範囲内の `Opacity` が漸減し、最終的に消える
4. 全汚れクリア or 制限時間で次フェーズへ
5. 最終リザルトで **スタイリッシュランク（C / B / A / S / SSS）** と **Sync Rate（2 人の協調度）** を表示

ルール上、**1P が撃たないと 2P がやることがない**、**2P が拭かないと 1P の画面が汚れていく**、という相互依存を明示的に組み込んでいる。

## 2. なぜ Unreal Engine 5.7 + C++ で作ったか

- **画面演出と物理が両立している既製品が必要だった**: トマトの軌道、跳ね返り、汚れマスクの拡散は Unity でも作れるが、UE のマテリアルとレンダーターゲットでビジュアル品質を稼げる方が「祭り感」を作りやすかった。
- **Slate / SWindow に手を入れる必要があると分かっていた**: スマホ側のサブ画面を Widget だけで作るのを早期に諦めたため（後述）、**`FSlateApplication::Get().AddWindow(...)` を素で叩ける UE の自由度** が決め手になった。
- **C++ + Blueprint 併用**: ゲームロジック・ハード入力・ウィンドウ管理は C++、レベル配置・パラメータ調整・演出フローは Blueprint、というよくある住み分け。`UPROPERTY(EditAnywhere)` で BP 側からチューニングできるよう積極的に露出した。

## 3. 一番の山: 「同じゲームの 2 つの画面」をどう出すか

### 3.1 ボツ案 ― シングルウィンドウ + Widget でスマホ枠を描く

最初は **OS から見たウィンドウは 1 つだけ** にして、その内側に Widget で「スマホ枠の絵」を描き、その上にズーム映像 RT を貼り付けていた。

実装としては素直で、ビルド設定もシンプル。**しかし以下の問題が次々に起きた**:

| 問題 | 症状 |
|------|------|
| ゲーム画面の引き伸ばし | モニターの解像度が変わるとメイン部分まで横方向にスケールされ、被写体の見え方が変わる |
| スマホ枠の位置ズレ | 物理的なスマホ実機を画面の左に置いて重ねる運用だったので、スマホの設置位置と Widget 内の枠位置を **環境ごとに合わせ込む必要** があった |
| 設定項目の爆発 | スマホ実機解像度・モニター解像度・スケーリング・スマホ位置オフセット…と環境ごとに 4〜5 個調整 |
| 試遊会での事故率 | 違うモニターに繋ぐたびにレイアウトが崩れ、開場直前に焦ってチューニングする時間が発生 |

「**ゲームを別の場所に持っていけるたびに 1 時間使う**」状態だったので、**1 つのウィンドウで 2 種類の構図を同居させる構成自体が筋悪** と判断した。

### 3.2 採用案 ― `SWindow` を 2 つ動的生成するデュアルウィンドウ

切り替え後の構成:

- メインウィンドウ: UE 標準のゲームウィンドウ。三人称視点のレンダリング。
- サブウィンドウ（スマホ画面）: 起動時に **C++ から `SWindow` を生成し `FSlateApplication` に `AddWindow`**。中身は `SImage` で、`RT_Zoom` の `UMaterialInstanceDynamic` を貼っているだけ。

擬似コード（`TomatinaHUD.cpp::CreatePhoneWindow` 周辺の要点）:

```cpp
TSharedRef<SWindow> PhoneWindow = SNew(SWindow)
    .Title(FText::FromString(TEXT("Tomatina Phone")))
    .ClientSize(FVector2D(PhoneWidth, PhoneHeight))
    .ScreenPosition(PhoneScreenPos)
    .SupportsMaximize(false)
    .SupportsMinimize(false)
    .UseOSWindowBorder(false)
    .SizingRule(ESizingRule::FixedSize);

PhoneWindow->SetContent(
    SNew(SImage).Image(&PhoneZoomBrush) // PhoneZoomBrush は DMI を参照
);

FSlateApplication::Get().AddWindow(PhoneWindow, /*bShowImmediately=*/true);
```

これによって得られたメリット:

- **OS 側で 2 ウィンドウとして配置できる** ので、スマホ実機（拡張ディスプレイ）にウィンドウをドラッグするだけで設置完了。
- 解像度や DPI が変わっても **メインの描画は一切影響を受けない**。引き伸ばしバグが消えた。
- 環境ごとに調整するパラメータが事実上 **「スマホ画面の横解像度 1 つ」** に集約された。
- スマホ枠の位置合わせが不要に（実機の物理的位置 = OS のディスプレイ配置で完結）。

### 3.3 なぜ nDisplay にしなかったか

UE には「複数ディスプレイに同じシーンを分散描画する」公式機能 nDisplay があり、当然候補に上がった。**最終的に採用しなかった理由**:

| 観点 | nDisplay | 本作の要件 |
|------|----------|-----------|
| 想定ユース | LED ウォール / CAVE / シミュレータなど **1 つのシーンを複数面でつなげる** | スマホには **別のカメラ視点（ズーム）** を出したい |
| カメラ構成 | 全ウィンドウが共通カメラの一部 | メインは三人称、スマホは `SceneCaptureComponent2D` の RenderTarget。**そもそも別シーン** |
| 学習・運用コスト | プラグイン有効化、ノード設定、Switchboard 含む | 学園祭規模の運用には重い |
| 環境依存 | コンフィグファイルの差し替えが前提 | 「USB さして起動するだけ」で済ませたかった |

要するに **「同じ世界の続きを 2 画面に分けたい」のが nDisplay、「別カメラの絵を 2 個並べたい」のが本作** で、目的がずれていた。`FSlateApplication::AddWindow` で素直に SWindow を 2 個出すのが、コード量・依存・運用すべてで軽かった。

## 4. デュアルウィンドウ化で芋づる式に出てきた問題と対処

ウィンドウを分離した直後は「真っ黒なスマホ画面」「ズームが時々止まる」「実機解像度ではアイコンが極小」など、別系統の問題が連鎖した。それぞれ対処したものを並べる。

### 4.1 DPI スケーリング: スマホ枠の中身が小さい / 大きい

UE の Slate は DPI スケーリングを掛けるが、**サブ `SWindow` 側の取得タイミングと Viewport の取得タイミングが噛み合わない**。`UWidgetLayoutLibrary::GetViewportScale()` で取った値で逆補正を掛け、`LayoutPhoneZoomImage` 側で実寸ピクセル指定に倒すことで解決した。

> 初回フレームで `GetCachedGeometry()` が 0 を返すケースがあり、1 フレーム待ちでフォールバックする実装が現在も入っている。

### 4.2 UE 5.7 の SceneCapture 自動キャプチャが効かない

`USceneCaptureComponent2D::bCaptureEveryFrame = true` でも、**何故かサブウィンドウ表示時に描画タイミングが噛まず、RT が更新されないことがあった**。バージョン依存の挙動と判断し、**`Tick` 内で明示的に `CaptureScene()` を呼ぶ** ようにして強制キャプチャに切り替えた。

```cpp
// TomatinaPlayerPawn.cpp 側、毎フレーム
if (ZoomCapture && ZoomCapture->TextureTarget)
{
    ZoomCapture->CaptureScene();
}
```

副作用として描画コストが上がるが、**毎フレーム確実にスマホ側へ最新の絵が出ること** が体験上必須なので受け入れた。

### 4.3 第二 SWindow で RenderTarget の Brush が再評価されない

`SImage` に Brush として渡したテクスチャが **メインウィンドウでは更新されるのに、サブウィンドウだけ初期フレームで止まる** 現象が発生した。原因は Slate がサブウィンドウの Brush を「変わっていない」と判定してテクスチャ参照を更新していなかったこと。

回避策として **`UMaterialInstanceDynamic` を介して RT を貼り、毎フレーム DMI のパラメータを再注入する** ことで「変化した」と判定させ、確実に再描画させている。

```cpp
// 毎フレーム
PhoneZoomDMI->SetTextureParameterValue(TEXT("ZoomRT"), ZoomRT);
PhoneZoomBrush.SetResourceObject(PhoneZoomDMI);
```

これは**“Slate の最適化を意図的に騙す”実装**で、本来は別解（毎フレーム新規 Brush を作る等）もあるが、**コスト・可読性のバランスでこの方式に落ち着いた**。

### 4.4 ズームカメラの壁抜け

ズーム時、被写体に寄ろうとするとカメラが壁を貫通して「内側」に入り、世界の裏側が見えてしまった。

対処は古典的だが堅実な組み合わせ:

1. **Sphere Sweep** で被写体までの間に障害物がないか判定
2. ヒットしたら手前で止める **Safety Margin** を加味
3. それでも貫通気味になる場合のため `CustomNearClippingPlane` を狭める

これで「街の建物にめり込まずに最大ズーム」が安定して取れるようになった。

### 4.5 空 / 何もない方向にカメラを向けた時の挙動

線分トレースが何にもヒットしないケース（空、街の外）では、距離が「無限」扱いになりカメラが暴走する。`SkyFallbackDistance` という値を持っておき、**ヒットしなかったら仮想的に N メートル先に当たったことにする** ロジックを入れて挙動を安定させた。

## 5. その他の技術的な見どころ

短くなるが、**ES の自由記述で 1〜2 個ずつピックして話すと刺さる** トピックを並べる。

### 5.1 `SetOverlayMaterial` で全キャラに汚れマスク（UE 5.0+）

トマトが当たった部分のキャラに「汚れ」を表現するため、各キャラの `USkeletalMeshComponent::SetOverlayMaterial` を使ってマスクマテリアルを乗せた。マテリアルパラメータは `ATomatoDirtManager` の `FDirtSplat` 配列に同期して、**Blueprint 側からも `UPROPERTY` で全部いじれる** ようにしてある。

### 5.2 Sticky Tomato（黄色） ― 「連続ダッシュ × 4」のミニゲーム

黄色いトマト（`EDirtType::StickyYellowDash`）は、ただ拭いても消えない。**短時間のうちに連続して大きく手を動かす（=ダッシュ）を 4 回成立させる必要がある**。

入力判定は `WipeDirtAt` の引数 `Amount` と前フレームからの移動距離で行い、

- `StickyDashMinAmount`（最小拭き取り量）
- `StickyDashMinMoveDistance`（最小移動距離）
- `StickyDashMaxInterval`（連続扱いになる最大間隔）

の 3 つで「ダッシュ判定」を組み立てている。ゲーム中は `StickyProgressMinOpacity` で進捗を視覚化し、**「もうちょっとで剥がれる」** をプレイヤーに伝えている。

### 5.3 ポーズ可能 UI: `SetGlobalTimeDilation(0)` + `FApp::GetDeltaTime()`

スコア集計時に世界全体を止めたいが、UI は動かしたい。`SetGlobalTimeDilation(0.f)` で世界を止め、UI 側のアニメーションは **`FApp::GetDeltaTime()` を直接拾って自前で進行** させている。Tick の `DeltaTime` を使うと一緒に止まるため、**「リアルタイム時間」を取りに行く** 区別を意識的にしている。

### 5.4 拭き取り SE のエッジ検出

拭く音をループ再生したいが、「拭いている間だけ」鳴らしたい。Wipe 入力の有無の **立ち上がり / 立ち下がりエッジ** を `UpdateWipeSound` で取り、Start / Stop を 1 回ずつ呼ぶ実装にした。地味だが「ずっと鳴り続けて煩い」を防ぐ要点。

### 5.5 BeginPlay 前のプロパティ注入

汚れの初期テクスチャや、ミッションごとの設定値を **`BeginPlay` よりも前に注入したい** ケースがあったため、`SpawnActorDeferred` でアクターを保留生成 → `UPROPERTY` を直接書き込んでから `FinishSpawning` を呼ぶ、というイディオムを各所で使っている。

## 6. 残課題 ― 一番大事な「正直に書く」セクション

ES でも Zenn でも、**残課題を伏せない** のが一番効くと考えている。

### 6.1 試遊会で出た UX 上の最大の課題

> **「1P がスマホ画面を見ながらメインモニターを操作するのが、想像以上に難しい」**

設計時は、**1P がスマホを“覗き込みながら”構図を決めて、メインに撃つ** ことを想定していた。しかし実際にプレイヤーに触ってもらうと、

- スマホは縦長で視野の端にあるため、**注視するためには首を大きく動かす必要がある**
- 操作中（トマト発射の照準）に 1P の視線がメインに固定されがち
- スマホ側を「あったら嬉しい補助情報」として無視してプレイされることが多かった

これは **ゲーム設計とハード設計の橋渡しが甘かった** 反省で、対策の方向性として:

1. スマホ画面に出す情報を **「常時見る」前提から「決定的瞬間にだけ見る」前提に切り替え** る
2. メインモニター側に **小さなピクチャ・イン・ピクチャ** でズーム映像を冗長表示する
3. 物理レイアウトとして **スマホをモニターに近づける** / 角度を起こす
4. もしくは「スマホを見たほうが得をする」報酬設計（ベストショット判定など）に倒す

の 4 案を比較中。**ハードの構成を変えるより、ゲームデザインで吸収する方向** が現実的だと考えている。

### 6.2 LeapMotion がパッケージビルドで認識されないケース

エディタ実行では問題ないが、`Tomato.exe` でパッケージ化したビルドで稀に認識しない事象を確認している。`Tomato.uproject` に `UltraleapTracking` プラグインを明示的に列挙する対応 + `TomatinaTowelSystem.cpp` 側に診断ログを仕込んだ状態。**次工程は再パッケージ + `Saved/Logs/Tomato.log` の解析**。

### 6.3 サブウィンドウの初期位置が手動

`PhoneScreenPos` を起動時に固定値で指定しているため、**展示先の OS のディスプレイ配置に依存** する。理想は「拡張ディスプレイのうち縦長のものを自動検出して、その左上に配置」ただし優先度低。

### 6.4 DPI キャッシュ初回 0 問題

`LayoutPhoneZoomImage` での `GetCachedGeometry()` 初回 0 を 1 フレーム遅延でしのいでいる。実害はないがコードの匂いは残っている。

## 7. 学んだこと（自由記述向け）

ES の「ガクチカ / 強み / 学んだこと」欄でそのまま使える粒度に整理する。

1. **設計を捨てる勇気を持てた**。最初の Widget 重ね方式は動いていたし、捨てるのは怖かったが、**「保守コストが指数的に上がる構造を放置する方が損失が大きい」** と判断して作り直したことで、結果的にデバッグ時間が激減した。
2. **公式機能（nDisplay）を“使わない”判断にも理由を言語化した**。デフォルト思考で機能を採用しに行くのではなく、**自分のユースケースを 1 行で書いてから候補と突き合わせる** クセがついた。
3. **UE のレイヤーを順に剥がしていけば直せると体感した**。最初は「Slate？SWindow？怖そう」だったが、コードを 1 段下に潜るだけで挙動が説明可能になり、**ブラックボックスを減らす** こと自体が一番のスキルアップだった。
4. **ハード × ゲームデザイン × UX の三点バランス** を、机上ではなく **試遊会の人の動きを見て** 学んだ。残課題 6.1 はこの学びの裏返しで、**「次に作るときは設置動線をペーパープロト段階で確かめる」** ことを次のプロジェクトの最優先事項に置いている。
